# 数据库表结构说明

数据库文件：`data/etf_manager.db`（SQLite，WAL模式），共 6 张表。

## 表命名规范

- **命名方式**：蛇形命名法（snake_case）
- **表名规则**：复数形式或名词形式，业务语义清晰
- **字段命名**：
  - 主键：表名单数_id（如 holding_id, record_id）
  - 时间：*_at（如 created_at）或 *_date（如 first_buy_date）
  - 布尔：is_*（如 is_active）
  - 金额：amount 或 *_amount
  - 份额：shares 或 *_shares
- **Schema 唯一权威来源**：`backend/models.py`（ORM 模型），`scripts/init_db.py` 与之完全对齐

## 2. fund_holdings — 持仓管理表

记录每支基金在不同平台的持仓情况，同一基金在不同平台各一条记录。
定投相关信息（频率、金额、类型等）已整合到本表中。
基金名称通过外键关联 `fund_info` 表获取，避免数据冗余。

| 字段 | 类型 | 说明 |
|------|------|------|
| holding_id | INTEGER | 持仓记录ID（主键，自增） |
| fund_code | TEXT | 基金代码（外键 → fund_info.fund_code），NOT NULL |
| platform | TEXT | 持仓平台（NOT NULL），如 支付宝/涨乐财富通 |
| holding_shares | REAL | 持有份额 |
| avg_buy_price | REAL | 平均买入价/成本价(元) |
| base_shares | REAL | 底仓份额（不可卖出的1/5仓位） |
| current_price | REAL | 净值/当前价格(元) |
| holding_value | REAL | 市值/持仓总值(元) |
| invested_capital | REAL | **累计买入总金额(元)**，只增不减 |
| total_sold | REAL | **累计卖出总金额(元)**，只增不减（v3新增） |
| net_invested | REAL | **净投入(元)** = invested_capital - total_sold（v3新增） |
| profit_loss_amount | REAL | 持有收益(元) = 市值 - 持有成本 |
| return_rate | REAL | 持有收益率(%) |
| dca_is_active | INTEGER | 定投是否启用（1=启用，0=停用） |
| dca_frequency | TEXT | 定投频率，如 每天定投100 |
| dca_amount | REAL | 每期金额(元) |
| dca_type | TEXT | 定投类型（固定金额/智能定投） |
| dca_total_invested | REAL | 定投累计金额(元) |
| first_buy_date | TEXT | 首次买入日期 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

> **唯一约束**：UNIQUE(fund_code, platform) — 同基金同平台只允许一条持仓记录
>
> **v3 变更**：
> - 新增 `total_sold`（累计卖出金额）和 `net_invested`（净投入）
> - 移除废弃的 `last_update_date` 字段（代码中只使用 `updated_at`）
> - `platform` 改为 NOT NULL（业务必填）
> - 添加 (fund_code, platform) 联合唯一约束

**计算公式**：
```
持有成本 = 持有份额 × 平均买入价（加权平均成本价）
持仓总值(市值) = 持有份额 × 当前净值
持有收益 = 持仓总值 - 持有成本
持有收益率 = 持有收益 ÷ 持有成本 × 100%
累计买入总金额 = 所有买入/定投交易的金额之和（卖出不扣减）
累计卖出总金额 = 所有卖出交易的回收金额之和
净投入 = 累计买入总金额 - 累计卖出总金额
```

**持仓重算机制**：
- 删除/修改已执行的交易记录时，自动触发 `rebuild_holding()` 函数
- 该函数按 (fund_code, platform) 分组，拉取全部已执行交易，从零回放计算持仓
- 采用"全量回放"策略而非"逆向差额"，确保数据一致性

## 3. daily_quotes — 每日净值表

存储每支基金的每日行情数据，用于生成收益曲线和触发交易信号。
基金名称通过外键关联 `fund_info` 表获取，避免数据冗余。

| 字段 | 类型 | 说明 |
|------|------|------|
| quote_id | INTEGER | 记录ID（主键，自增） |
| fund_code | TEXT | 基金代码（外键 → fund_info.fund_code） |
| quote_date | TEXT | 净值日期，格式 YYYY-MM-DD |
| open_price | REAL | 开盘价（场内ETF）/净值（场外基金） |
| high_price | REAL | 最高价 |
| low_price | REAL | 最低价 |
| close_price | REAL | 收盘价/单位净值 |
| acc_nav | REAL | 累计净值 |
| created_at | TEXT | 创建时间 |

> **唯一约束**：(fund_code, quote_date)

---

## 4. trading_rules — 交易规则配置表

将交易规则结构化存储，用于自动生成交易信号。

| 字段 | 类型 | 说明 |
|------|------|------|
| rule_id | INTEGER | 规则ID（主键，自增） |
| fund_category | TEXT | 适用基金分类，如 军工/芯片/黄金 |
| rule_type | TEXT | 规则类型：止盈/接回/大跌加仓/核心纪律 |
| condition_desc | TEXT | 触发条件描述 |
| threshold | REAL | 阈值（如涨跌幅百分比） |
| action_desc | TEXT | 触发后的操作描述 |
| priority | INTEGER | 优先级（数字越大越优先） |
| is_active | INTEGER | 是否启用（1=启用，0=停用） |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

## 5. trade_records — 交易记录表

统一记录所有交易操作和信号，合并了原来的 transactions（交易流水）和 trade_signals（交易信号）。
基金名称通过外键关联 `fund_info` 表获取，避免数据冗余。

| 字段 | 类型 | 说明 |
|------|------|------|
| record_id | INTEGER | 记录ID（主键，自增） |
| fund_code | TEXT | 基金代码（外键 → fund_info.fund_code），NOT NULL |
| record_type | TEXT | 记录类型：买入/卖出/定投/信号 |
| record_date | TEXT | 记录日期 |
| signal_type | TEXT | 信号类型：止盈/接回/大跌加仓 |
| trigger_condition | TEXT | 触发条件描述 |
| trigger_value | REAL | 触发时的数值 |
| suggested_action | TEXT | 建议操作 |
| exec_status | TEXT | 执行状态：待执行/已执行/已忽略 |
| exec_date | TEXT | 执行日期 |
| actual_action | TEXT | 实际执行的操作 |
| platform | TEXT | 持仓平台 |
| amount | REAL | 交易金额(元) |
| shares | REAL | 交易份额 |
| nav | REAL | 成交净值 |
| fee | REAL | 手续费，默认0 |
| note | TEXT | 备注 |
| created_at | TEXT | 创建时间 |

> **record_type 说明**：
> - `买入`/`卖出`/`定投`：实际发生的交易，会联动更新 fund_holdings
> - `信号`：由规则引擎自动生成的待执行信号，不自动更新持仓
>
> **联动行为**：
> - 创建记录时：若 record_type 为买入/卖出/定投，调用 `apply_transaction()` 更新持仓
> - 删除/修改记录时：若涉及已执行的实际交易，调用 `rebuild_holding()` 全量回放重算持仓

## 6. portfolio_snapshots — 每日资产快照表

记录每个交易日的投资组合整体资产状况，用于收益曲线展示和历史回溯。
通过回放 `trade_records` 交易记录和 `daily_quotes` 净值数据自动生成。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 快照ID（主键，自增） |
| snapshot_date | TEXT | 快照日期（NOT NULL，UNIQUE），格式 YYYY-MM-DD |
| total_assets | REAL | 当日总资产/总市值(元)，DEFAULT 0 |
| total_invested | REAL | 累计买入总金额(元)，DEFAULT 0 |
| total_pnl | REAL | 浮动盈亏 = 市值 - 持有成本，DEFAULT 0 |
| pnl_rate | REAL | 收益率(%) = 浮动盈亏 ÷ 持有成本 × 100，DEFAULT 0 |
| realized_pnl | REAL | 已实现收益(元)，卖出时锁定的盈亏，DEFAULT 0 |
| fund_count | INTEGER | 当日持仓基金数量，DEFAULT 0 |
| created_at | TEXT | 创建/更新时间 |

> **唯一约束**：snapshot_date（每天只有一条汇总快照）
>
> **生成方式**：运行 `scripts/generate_snapshots.py`
> - `python scripts/generate_snapshots.py` — 补录全部历史快照
> - `python scripts/generate_snapshots.py today` — 仅生成/更新今日快照
>
> **计算逻辑**：
> - 按时间顺序回放所有已执行的交易记录，逐日累积持仓份额
> - 买入/定投：加权平均计算成本价，累计买入金额只增不减
> - 卖出：减少份额，按卖出份额×成本价计算已实现收益
> - 每日市值 = 各基金持仓份额 × 当日净值（无净值时用最近历史净值兜底）
> - 浮动盈亏 = 市值 - 持有成本（份额 × 加权成本价）

## 表关系

```
fund_info (fund_code)  [主表]
  ├── fund_holdings (fund_code) ── 1:N，一支基金可在多个平台持仓
  │     └── UNIQUE(fund_code, platform)，同基金同平台只允许一条
  ├── daily_quotes (fund_code) ── 1:N，一支基金有多天净值
  └── trade_records (fund_code) ── 1:N，一支基金有多条交易/信号记录

trading_rules ── 独立表，通过 fund_category 字段与 fund_info.fund_category 逻辑关联

portfolio_snapshots ── 独立汇总表，通过回放 trade_records + daily_quotes 自动生成
                       不直接关联 fund_info，记录的是整个投资组合的每日快照
```

> **数据规范化说明**：子表（fund_holdings、daily_quotes、trade_records）不再冗余存储 `fund_name` 字段，
> 改为通过外键 `fund_code` 关联 `fund_info` 表动态获取，确保数据一致性。

## 数据库维护建议

1. **定期备份**：每周自动备份数据库
2. **索引优化**：定期检查索引使用情况
3. **数据清理**：定期清理历史交易记录和行情数据
4. **版本管理**：数据库结构变更需记录版本号
5. **快照更新**：每日拉取净值后运行 `generate_snapshots.py today` 更新当日快照
6. **Schema 来源**：所有表结构以 `backend/models.py` 为唯一权威定义，`init_db.py` 和文档需与之同步
